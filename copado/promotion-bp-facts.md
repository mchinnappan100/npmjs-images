Fact Check: Copado Promotion and Conflict ManagementThis report validates the technical claims regarding the reuse of Promotion records and the preservation of deployment state in Copado.1. Claim: "A promotion record acts as the container for deployment state, including resolved merge conflicts."Status: VERIFIED (With Nuance)Technical Detail: In Copado, a Promotion record is linked to a specific Git branch (the "Promotion Branch"). When Copado merges User Stories into this branch, any manual conflict resolutions performed via the Copado "Conflict Resolution" interface are technically stored in a dedicated Conflict Resolution branch (usually prefixed with ccr/).The Container: While the Git branch holds the code, the Promotion Record serves as the orchestrator. If the record is deleted or bypassed, the link between the User Stories and that specific "state" of the merged code is severed.2. Claim: "Creating a new promotion record resets this state, which leads to repeated work."Status: CORRECTTechnical Detail: By default, when you create a New Promotion for the same set of User Stories, Copado attempts a fresh merge from the feature branches of those stories into a brand-new Promotion Branch.The Impact: If conflicts were previously resolved in an older Promotion, and those resolutions were not committed back to the base branches (which they rarely are), the developer must re-resolve those same conflicts in the new Promotion. This is a common source of "merge fatigue" in Copado implementations.3. Claim: "By reusing the same promotion record and attaching a new promotion branch, teams maintain continuity."Status: TECHNICALLY FEASIBLE / ARCHITECTURAL BEST PRACTICEThe Mechanism: Instead of creating a new record, teams can "re-promote" or "re-deploy" using the same record. In Copado's advanced settings, you can point a Promotion to an existing branch or reuse the one already created.Efficiency: Reusing the record (and thus the branch) ensures that the "Conflict Resolution" records associated with that Promotion are maintained. You are effectively "appending" or "refreshing" the existing merge rather than starting from scratch.4.
